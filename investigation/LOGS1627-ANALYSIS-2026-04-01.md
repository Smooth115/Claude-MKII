## 2. NVMe PCI SLOT SWAP — HYPERVISOR FINGERPRINT (F1, F2)

### Boot -6 (fake Aug 2024 timestamp):
```
nvme nvme0: pci function 0000:05:00.0 → nvme0n1: p1 p2 p3
nvme nvme1: pci function 0000:02:00.0 → nvme1n1: p1
EXT4-fs (nvme0n1p2): UUID b9dfb7d1-85a4-4208-b8b0-4109685553a0
```

### Boots -5 to -3 (Mar 28 02:16–02:23):
```
nvme nvme0: pci function 0000:02:00.0 → nvme0n1: p1 (only!)
nvme nvme1: pci function 0000:05:00.0 → nvme1n1: p1 p2 p3
EXT4-fs (nvme1n1p2): UUID b9dfb7d1-85a4-4208-b8b0-4109685553a0 ← SAME UUID, different device
```

### Boot 0 (Mar 28 15:55, final):
```
nvme0n1: p1 p2 p3
EXT4-fs (nvme0n1p2): UUID b9dfb7d1-85a4-4208-b8b0-4109685553a0 ← back to nvme0
```

**Interpretation:** The same physical data (UUID b9dfb7d1) appears on `nvme0n1p2` in some boots and `nvme1n1p2` in others. The PCI function addresses also swap (0x05:00.0 ↔ 0x02:00.0). This is the hypervisor remapping virtual PCI slots between boots — the guest OS sees a different device order but the underlying data is the same.

This directly corroborates TheLink.txt's virtual dmar1 finding and the SubVirt/Blue Pill hypervisor architecture already documented.

## 3. DMAR — VM FIRMWARE PROVIDER (F5)

```
ACPI: DMAR 0x000000006D304000 0000A8 (v01 INTEL  EDK2  00000002  01000013)
ACPI: DMAR 0x000000006D305000 0000A8 (v01 INTEL  EDK2  00000002  01000013)
DMAR: [Firmware Bug]: No firmware reserved region can cover this RMRR
DMAR: [Firmware Bug]: Your BIOS is broken; bad RMRR [0x000000003e2e0000-0x000000003e2fffff]
```

**Two DMAR tables at adjacent memory addresses**, both with provider string `EDK2`. Real ASUS BIOS hardware would show a single DMAR with ASUS/AMI provider string. EDK2 is TianoCore/OVMF — the open-source UEFI firmware used in virtual machines.

The "bad RMRR" Firmware Bug is the kernel complaining that the fake ACPI tables have inconsistent memory reservations. The hypervisor's ACPI emulation is imperfect.

**Cross-reference:** BIOS reports `American Megatrends v2.7` but DMAR tables say `EDK2`. These are inconsistent and confirm the ACPI tables are being spoofed by the hypervisor layer.

## 5. XEN DIRECTORY ON WINDOWS NTFS PARTITION (F4)

```
2026-03-28T00:37:22 udisksd: Mounted /dev/nvme1n1p1 at /media/lloyd/yoink
2026-03-28T00:37:33 kernel: ntfs3: nvme1n1p1: ino=62308, "xen" directory corrupted
2026-03-28T00:37:33 kernel: ntfs3: nvme1n1p1: Mark volume as dirty due to NTFS errors
(message repeated 13+ times over ~25 minutes)
2026-03-28T02:16:39 kernel: /dev/nvme1n1p1: 11 files, 1572/274630 clusters
```

The Windows/NTFS partition (`nvme1n1p1`, mounted at `/media/lloyd/yoink`) contains a directory named `"xen"` at inode 62308. **Xen is a type-1 hypervisor.** This directory is "corrupted" per the NTFS driver — meaning its on-disk structures are inconsistent, possibly because the hypervisor itself is writing to that partition in ways the NTFS driver can't reconcile.

The `yoink` mount point name appeared in TheLink.txt and was initially thought to be user files. The logs confirm it's the label udisks assigned to the Windows NTFS partition — it's not a user file, it's the Windows drive.

11 files, only 1572 of 274,630 clusters used = effectively empty except for those 11 files. The "xen" directory is likely part of the attacker's hypervisor deployment structure.

## 8. PAM_LASTLOG.SO MISSING (F7)

```
2026-03-28T01:14:41 login[658851]: PAM unable to dlopen(pam_lastlog.so): 
  /usr/lib/security/pam_lastlog.so: cannot open shared object file: No such file or directory
2026-03-28T01:14:41 login[658851]: PAM adding faulty module: pam_lastlog.so
```

`pam_lastlog.so` is the PAM module that records the last login time (displayed at login: "Last login: ..."). Its absence means:
1. Login history tracking is disabled
2. The `/var/log/lastlog` database is not being updated
3. Attacker access timestamps are not recorded

In Ubuntu 24.04, `pam_lastlog.so` is part of `libpam-modules`. Its absence on a fresh install points to deliberate removal or replacement of PAM configuration.
